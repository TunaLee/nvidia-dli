# 파이널 프로젝트 — LLM 에이전트 호스트 웹앱

주제는 자유다. LLM 에이전트를 탑재한 웹 앱을 만들고, 시연으로 채점한다.

## 필수 요건

아래 두 가지가 기본 점수의 범위다. 이 안에서의 완성도를 채점한다.

**1. 웹 앱**

- 프레임워크 자유 (수업 기준: Next.js)
- 채팅 또는 그에 준하는 상호작용 UI
- LLM 응답 **스트리밍** 표시
- 로딩·에러 상태 처리 (API 실패 시 빈 화면이 아닐 것)

**2. LLM 에이전트**

- 판단 → 도구 → 관찰 루프 (7강 구조)
- **직접 정의한 도구 1개 이상** — LLM이 상황에 맞게 도구를 골라 호출해야 한다
- 모델: NVIDIA API (Qwen) 기준, 다른 관리형·자가 서빙도 허용
- API 키는 환경변수로 — 코드에 하드코딩하면 감점

## 가점 항목

각 항목 +5점, **최대 +20점**. 시연에서 실제로 동작해야 인정된다.

| 항목 | 인정 기준 |
|---|---|
| MCP 서버 | 직접 구현한 MCP 서버를 앱이 도구로 연동 |
| 멀티에이전트 | LangGraph 병렬(Send) 또는 역할 분담 오케스트레이션이 실제 기능에 쓰임 |
| DB 저장 | 대화·데이터가 DB(Neon 등)에 저장되고 재접속 시 유지 |
| 소셜 로그인 | Google OAuth 등으로 사용자 구분 |
| 웹검색 도구 | Tavily 등 외부 검색이 에이전트 도구로 동작 |
| 배포 | Vercel 등 공개 URL에서 동작 |
| 하이브리드 서빙 | 역할별로 관리형·자가 서빙(Ollama)을 나눠 호출 |

네 개를 넘게 해도 +20점이 상한이다 — 개수보다 완성도에 시간을 쓰는 편이 유리하다.

## 채점 기준 (기본 100점 + 가점)

| 항목 | 배점 | 보는 것 |
|---|---|---|
| 동작·완성도 | 30 | 시연에서 핵심 흐름이 끝까지 도는가 |
| 에이전트 구현 | 25 | 도구 설계와 선택 정확성, 루프·프롬프트 품질 |
| 현실성 | 15 | 실제 문제를 푸는가, 누가 쓸 물건인가 |
| 창의성 | 15 | 주제와 접근의 신선함 |
| 디자인·UX | 15 | 스트리밍 표시, 로딩·에러 처리 포함한 사용 경험 |

시연 후 **아키텍처 문답**이 있다 — "이 요청이 들어오면 어디를 거쳐 응답까지 오는가"를 본인이 설명할 수 있어야 한다. 문답 결과는 동작·완성도 점수에 반영된다.

## 진행 방식

1. **주제 피칭** — 시작 후 이른 시점에 주제와 핵심 기능 한 가지를 강사에게 승인받는다. 범위가 크면 여기서 줄인다.
2. **개발** — 코드 에이전트(Claude Code · Codex) 사용을 권장한다. 8~9강에서 만든 앱에서 출발해도 되고 새로 시작해도 된다. 채점 대상은 코드가 아니라 **동작하는 프로덕트와 본인의 이해**다.
3. **시연** — localhost 시연 허용. 만일에 대비해 핵심 흐름의 화면 녹화를 백업으로 준비한다.

## 주제 예시

그대로 만들어도 감점은 없다. 백지에서 고민하는 시간을 아끼는 용도다.

- 여행 플래너 — 웹검색으로 일정·명소를 모아 계획표 생성
- 가계부 비서 — 지출 내역을 DB에 쌓고 자연어로 조회·분석
- 회의록 정리기 — 녹취 텍스트에서 결정사항·할 일 추출
- 뉴스 리서처 — 주제를 주면 검색·요약·비교 보고서 작성
- 레시피 도우미 — 재료 기반 추천과 장보기 목록 생성
- 코드 리뷰 봇 — 저장소 파일을 읽고 리뷰 코멘트 생성

## 주의

- NVIDIA API rate limit — 시연 직전 리허설 호출을 아낀다. 한도 문제 시 강사에게 백업 엔드포인트를 요청한다.
- 시연 도중의 실패는 감점이지만, 에러를 처리해 보여주는 것(친절한 실패)은 완성도로 인정한다.
