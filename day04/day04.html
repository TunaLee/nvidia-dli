01
CHAPTER 01

Resource란
다시 깊게

읽기전용·URIstaticdynamic·템플릿
/
CH1본질

Resource = GET 같은 ‘읽기 창구’

→
읽기 전용 — 상태를 바꾸지 않는다(부작용 없음)
→
URI로 주소 지정 — docs://policy/leave 처럼
→
모델이 필요할 때 읽어 컨텍스트에 넣는다

비유

도구(Tool)가 ‘버튼’이라면, 리소스(Resource)는 ‘서랍 속 문서’ — 열어서 읽을 수 있고, 건드려도 바뀌지 않는다.

/
CH1두 종류

고정 리소스 vs 동적(템플릿) 리소스

static

고정 URI

config://status 처럼 하나의 고정 주소. 항상 같은 종류의 데이터를 반환.

dynamic

URI 템플릿

docs://{id} 처럼 매개변수를 받아 그에 맞는 데이터를 반환.

감각
템플릿은 ‘REST 경로 파라미터’와 같다 — {id}가 함수 인자로 들어온다.
/
CH1코드

URI 템플릿이 함수 인자로

template.py
DOCS = {"leave": "연차는 3영업일 전 신청", "expense": "비용은 영수증 첨부"}

@mcp.resource("docs://{doc_id}", description="사내 정책 문서를 ID로 읽는다")
def get_doc(doc_id: str) -> str:
    return DOCS.get(doc_id, "문서를 찾을 수 없습니다.")
# docs://leave 를 읽으면 → "연차는 3영업일 전 신청"
{doc_id} 자리표시자가 그대로 함수 인자가 된다

언제 Tool

읽기 vs 행동경계 사례
/
CH2경계 정리

경계가 애매할 때의 기준

상황선택이유
정책 문서를 읽기Resource읽기 전용·URI로 지정
문서를 새로 저장Tool상태 변경(부작용)
ID로 사용자 정보 조회Resource읽기·매개변수 → 템플릿
‘검색’해서 결과 반환상황에 따라단순 조회=Resource, 복잡 질의·연산=Tool
팁
‘검색’은 흔히 Tool로 둔다 — 질의 처리·랭킹 같은 ‘연산’이 들어가기 때문.
/
미니 실습설계

직접 해보기 — 리소스/툴 + URI 짓기

과제 — 무엇으로 만들지 정하고, Resource면 URI를 설계

A
“제품 코드로 사양서 보기” → ?
B
“오늘 회의 일정 잡기” → ?
C
“부서별 인원 현황 조회” → ?

예시 답

A=Resource specs://{code} · C=Resource org://dept/{name} · B=Tool(일정 ‘등록’=부작용).

CHAPTER 03

좋은 리소스의
설계 원칙

URI 스킴설명·MIME목록
/
CH3원칙

리소스를 잘 설계하는 법

일관된 URI 스킴

type://path/{param} 형태로 예측 가능하게. 종류별 접두사를 통일.

설명을 명시

description=로 무엇을 반환하는지 분명히 — 클라이언트 목록에서 보인다.

MIME 타입

텍스트·JSON 등 형식을 분명히 해 클라이언트가 잘 처리하게.

목록 제공

어떤 리소스가 있는지 ‘목록’으로 노출해 발견 가능하게.

/
CH3가장 흔한 실수

description= 빠뜨리면 생기는 일

✕ description 없음

docstring 파싱이 안 잡혀 클라이언트 목록에서 설명이 빈칸 → 모델이 언제 쓸지 모름.

✓ description 명시

“무엇을·언제 읽는 리소스인지” 한 줄 → 모델이 정확히 호출.

규칙
리소스엔 항상 description=를 직접 적는다(FastMCP의 docstring 파싱은 신뢰 불가).

CH4파일

폴더의 파일을 ID로 읽기

files.py
from pathlib import Path
BASE = Path("./docs")

@mcp.resource("docs://{name}", description="docs 폴더의 텍스트 파일을 읽는다")  # file://은 호스트로 해석돼 깨짐
def read_file(name: str) -> str:
    p = (BASE / name).resolve()
    if BASE.resolve() not in p.parents:   # 경로 탈출 방지(검증!)
        raise ValueError("허용되지 않은 경로")
    return p.read_text(encoding="utf-8")

보안

경로 매개변수는 반드시 검증 — ../ 같은 탈출을 막는다.

/
CH4표 데이터

CSV·DB 한 행을 리소스로

data.py
import csv
ROWS = {r["id"]: r for r in csv.DictReader(open("products.csv"))}

@mcp.resource("product://{pid}", description="제품 ID로 사양 행을 반환")
def product(pid: str) -> dict:
    return ROWS.get(pid, {"error": "not found"})

# DB도 동일 — 함수 안에서 SELECT 후 dict 반환(읽기 전용)
읽기 전용 SELECT는 Resource, INSERT/UPDATE는 Tool

lab2_dynamic.py
POLICY = {"leave": "연차는 3영업일 전 신청·승인",
          "expense": "비용은 영수증 첨부 후 팀장 승인"}

@mcp.resource("policy://{topic}", description="주제별 사내 정책을 반환")

LAB 3발견 가능성

무엇이 있는지 목록으로

@mcp.resource("policy://index", description="이용 가능한 정책 주제 목록")
def policy_index() -> list[str]:
    return list(POLICY.keys())   # ["leave","expense"]

CH7핵심 정리

오늘 꼭 가져갈 5가지

1

Resource = 읽기 전용·URI

GET 같은 ‘읽기 창구’.

2

static vs 템플릿

{param}이 함수 인자로 들어온다.

3

description=는 필수

안 적으면 설명이 빈칸 → 오용.

4

파일·CSV·DB를 리소스로

경로·입력은 반드시 검증.

5

읽기=Resource, 행동=Tool

검색처럼 연산이 들면 Tool로.

/
CH7오개념 정리

헷갈리기 쉬운 지점

오해사실
“리소스도 데이터를 바꿔도 됨”읽기 전용 — 변경은 Tool
“URI는 아무렇게나”일관된 스킴이 모델 활용도를 높인다
“경로는 그대로 열면 됨”경로 검증 없으면 탈출·유출 위험
“검색은 당연히 Resource”질의 연산이 크면 Tool이 적합
