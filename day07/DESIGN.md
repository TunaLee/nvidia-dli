## 신규 랩의 근거

### Lab 6 · 진짜 RAG를 도구로

Day02 노트북은 이렇게 끝난다:

> 💡 이 `advanced_rag()`가 Week 3에서 **에이전트의 검색 도구**가 됩니다.

그런데 Day07의 `search_docs`는 하드코딩된 문장 몇 개를 문자열 매칭하는 장난감이었다. **커리큘럼이 스스로 한 약속을 지키는 랩이다.**

- **벡터 검색만** 쓴다 (하이브리드·리랭커 제외). 임베딩 모델 하나면 되고, Day01에서 이미 받아놔 HF 캐시에 있다.
- 도구 **이름·시그니처는 그대로**, 속만 바꾼다 → 에이전트 코드는 한 줄도 안 바뀐다는 걸 보여준다.
- **Agentic RAG**: Day01~02에선 검색이 *항상* 돌았지만, 이제 **에이전트가 검색할지 말지 판단**한다.
  - `"2+2는?"` → 도구 호출 0회
  - `"연차 며칠 전에 신청해?"` → `search_docs` 호출
  - Day02 제목이 "Agentic RAG"인데 정작 그 체감이 없던 걸 여기서 채운다.

### Lab 7 · 미니 코드 에이전트

**핵심 메시지: 코드 에이전트는 마법이 아니다.** 같은 루프에 코드 도구를 물린 것뿐이다. 도구 4개(`list_files`·`read_file`·`write_file`·`run_python`) 약 25줄이면 끝난다.

세 단계로 진행한다:

1. **버그 수정 (RED → GREEN)** — 실패하는 테스트를 통과시킨다. Lab 4의 self-correction이 그대로 재등장.
2. **`run_python`을 빼고 같은 과제** ⭐ — 에이전트가 코드를 고치고 "완료했습니다"라고 답하지만 **검증을 못 한다.** 실행 도구를 다시 주면 스스로 실패를 보고 고친다.
   → **왜 코드 에이전트에 실행·검증 도구가 필수인가** = 챗봇과 코드 에이전트를 가르는 선.
3. **샌드박스 경계** — 밖의 경로를 읽으라 하면 가드가 막고, 그 에러가 되먹여진다.
   → Lab 9(신뢰 경계 비교)로 넘어가는 다리.

**한계까지 부딪히게 하는 게 목적이다.** `write_file`은 파일을 통째로 덮어쓴다 → 큰 파일에서 망가진다. 컨텍스트도 금방 찬다. 가드는 경로 검사 한 줄뿐이다. 이 격차를 메우는 게 **하네스(Day08)** 다.

> 작동하는 코드 에이전트는 쉽고, **믿을 수 있는** 코드 에이전트는 어렵다.

### Lab 8 · 모델 스위칭 벤치

같은 루프·같은 과제에 `LLM_MODEL`만 바꾼다. 배우는 것:

1. **루프는 모델과 무관하다** — 같은 코드가 전부 돌아간다
2. 작은 모델은 도구를 엉뚱하게 부르거나 루프를 못 끝내고 `max_steps`에 걸린다
3. **코드 에이전트에선 모델 선택이 곧 성능이다**

모델별 tool-calling 지원 여부는 사전 검증하지 않았다. 랩이 **하나씩 시도하고 실패하면 `도구 미지원`으로 표시**하도록 짠다 → 안 되는 모델도 교보재가 된다.

### Lab 9 · Claude Code / Codex 병행 비교

Lab 7과 **똑같은 과제**를 각자 구독한 제품에 시킨다. (수강생은 둘 중 **하나만** 구독한다고 가정 → 서로 다른 걸 고른 사람끼리 비교)

두 제품을 가르는 축은 **신뢰 경계를 어디에 두느냐**다:

| | Claude Code | ChatGPT Codex |
|---|---|---|
| 기본 실행 위치 | 내 머신 (로컬 우선) | 클라우드 샌드박스 (repo 클론) |
| **신뢰 경계** | **사람이 게이트** — 권한 모드 승인 | **환경이 게이트** — OS 커널 샌드박스 |
| 상호작용 | 대화형, 중간에 끼어들어 조종 | 과제 위임 → diff·로그·PR 수령 (비동기) |
| 프로젝트 지시 | `CLAUDE.md` | `AGENTS.md` |

Lab 7에서 샌드박스를 손으로 짜본 직후라, 이 대비가 그대로 꽂힌다.
