관리형 API의 역할

→
NVIDIA API는 서빙을 NVIDIA가 대신 운영하는 형태다 (NVIDIA 자체 서빙 인프라). 사용자에게는 OpenAI 호환 /v1 규격만 노출된다.
→
OpenAI · Anthropic API도 구조는 같다 — 관리형 서빙이다.
★
관리형 API의 대가는 호출량 제한(rate limit)이다. 동시성 캡·백오프는 우회책이고, 자가 서빙이 근본 해법이다.
실제 호출 — 노출되는 것은 이것뿐
POST https://integrate.api.nvidia.com/v1/chat/completions
Authorization: Bearer nvapi-...

{ "model": "qwen/qwen3-next-80b-a3b-instruct",
  "messages": [{"role":"user","content":"..."}],
  "stream": true }

# 이 호출 뒤에서 배칭 · KV 캐시 ·
# GPU 스케줄링을 서빙 엔진이 처리한다
정리
관리형 API는 서빙을 대신하는 대신 호출량을 제한한다. 같은 요청이 자가 서빙(localhost)에도 그대로 동작한다.

CH1배경

생성이 느린 구조적 이유

→
LLM은 오토리그레시브 — 토큰을 하나 만들 때마다 지금까지의 전체 문맥을 입력으로 다시 계산한다.
→
1,000토큰 답변 = forward 1,000번.
✕
순진하게 구현하면 매번 같은 앞부분을 다시 계산하게 된다.
생성 루프의 구조
문맥: [나는] → 다음 토큰 '오늘'
문맥: [나는, 오늘] → '서빙'
문맥: [나는, 오늘, 서빙] → '을'
...

# 매 단계의 attention 계산이
# 앞 토큰 전체를 다시 본다
정리
이 반복을 줄이기 위해 앞 토큰들의 attention 중간 계산(K·V)을 저장해 재사용한다 — 이것이 KV 캐시이고, 서빙 엔진 최적화의 출발점이다.
/
CH1원리 1

계산 재사용

→
attention에서 각 토큰의 K(key)·V(value)는 한 번 계산되면 변하지 않는다.
→
그래서 저장해 두고 재사용한다 — 새 토큰 하나 분량만 계산하면 된다.
★
생성 속도가 극적으로 빨라지는 대신, 메모리를 먹기 시작한다.
→
KV 캐시 크기는 문맥 길이 × 레이어 × 헤드에 비례한다.
→
긴 대화 · 동시 요청이 많을수록 GPU 메모리를 잠식한다.
✕
8강의 '컨텍스트는 유한 자원'이 서빙 쪽에서는 메모리 문제로 나타난다.
정리
KV 캐시는 계산량을 메모리 사용량으로 바꾸는 기법이다. 이후 과제는 메모리 관리다.
/
CH1문제

예약 방식의 비효율

순진한 할당
요청 A: [========........]  최대 길이만큼 예약
요청 B: [=====...........]  실제론 일부만 사용
요청 C: [==============..]

# 쓰지도 않을 뒷부분을 미리 잡아둔다
# 조각난 빈틈은 다른 요청이 못 쓴다
✕
요청마다 최대 길이를 통째로 예약 — 안 쓰는 부분이 낭비된다.
✕
요청이 끝나고 남은 자리가 조각나 새 요청이 들어오지 못한다.
→
GPU 메모리의 상당 부분이 실제로는 비어 있는 채 낭비됐다.
결과
메모리가 남는데도 동시 요청을 더 받지 못한다 — 처리량이 메모리 관리에 막히는 상황이다.

CH1원리 2

OS 페이징의 차용 — vLLM의 핵심

→

CH1원리 3

빈자리의 즉시 충전

static batching
배치: [A, B, C] 동시 시작
A 끝남 → 자리 비움 (대기)
B 끝남 → 자리 비움 (대기)
C 끝날 때까지 전원 대기
→ 다음 배치 [D, E, F] 시작
continuous batching
배치: [A, B, C]
A 끝남 → 그 자리에 D 즉시 투입
B 끝남 → E 즉시 투입

# 토큰 단위로 스케줄링
# GPU가 놀지 않는다
효과
요청 길이가 제각각인 실제 트래픽에서 처리량 차이를 만드는 핵심 기법이다. vLLM · SGLang(둘 다 GPU 서빙 엔진) 모두 기본 동작이다.

CH4설계

맵리듀스 모양의 오케스트레이터

그래프 구조
START
  |
[planner]   목표 → 작업 3개로 분해
  |  (팬아웃 — 작업 수만큼 Send)
[worker] [worker] [worker]   ← 병렬
  \      |      /
   [aggregate]   결과 종합
        |
       END
→
planner — LLM이 목표를 독립 작업으로 분해한다.
→
Send 팬아웃 — 작업 수만큼 worker 노드를 동적으로 띄운다. 같은 스텝 안에서 병렬로 돈다.
→
aggregate — reducer로 모인 결과를 종합한다.
★

공유 상태와 분해 노드

state_and_planner.py
from typing import Annotated, TypedDict
import operator
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send

class State(TypedDict):
    goal: str
    tasks: list[str]
    results: Annotated[list[str], operator.add]  # 병렬 결과를 이어 붙이는 reducer
    final: str

def planner(state: State):
    out = ask(f"목표를 독립 작업 3개로 나눠라. 한 줄에 하나씩.\n목표: {state['goal']}",
              system="계획만 세운다. 실행하지 않는다.")
    tasks = [l.strip('-· ').strip() for l in out.splitlines() if l.strip()][:3]
    return {"tasks": tasks}
포인트
results의 reducer(operator.add)가 핵심이다. 병렬 워커들의 반환이 덮어쓰이지 않고 합쳐진다 — 8강 컨텍스트의 add_messages와 같은 원리다.

CH4코드

Send로 만드는 동적 병렬

fanout_and_worker.py
class WorkerState(TypedDict):
    task: str

def fan_out(state: State):
    # 작업 수만큼 worker 를 띄운다 — 이 리스트가 곧 병렬이다
    return [Send("worker", {"task": t}) for t in state["tasks"]]

def worker(state: WorkerState):
    ans = ask(f"다음 작업을 수행하라: {state['task']}")
    return {"results": [f"[{state['task']}]\n{ans}"]}
Send
조건부 엣지가 Send 목록을 반환하면, 각 항목이 독립 상태로 병렬 실행된다. 프롬프트 지시 없이 병렬이 보장된다.

컴파일과 동시성 제한

graph_build.py
g = StateGraph(State)
g.add_node("planner", planner)
g.add_node("worker", worker)
g.add_node("aggregate", aggregate)

g.add_edge(START, "planner")
g.add_conditional_edges(
    "planner", fan_out, ["worker"])
g.add_edge("worker", "aggregate")
g.add_edge("aggregate", END)

app = g.compile()

많이 쓰는 오케스트레이션 유형

유형구조단일 대비 이점
리서치 · 보고서 생성조사 → 분석 → 작성 → 검토역할별 프롬프트 특화 · 교차 검토
문서 일괄 분석 (리뷰·계약서·이력서)분할 → 병렬 분석 → 종합N건 병렬로 처리 시간 단축
코드 리뷰 패널보안·로직·스타일 관점 병렬 + 종합자기검열 사각 제거
고객 문의 처리분류 라우팅 → 전담 에이전트문의 유형별 정확도
콘텐츠 파이프라인목차 → 섹션 병렬 작성 → 편집섹션 병렬 · 톤 일관성은 편집 단계

CH5정리

오늘 채운 층

1

서빙의 자리

API 뒤에서 모델을 GPU에 올리고 요청을 받아내는 층이다. 관리형 API는 이 층을 대신해 주는 대신 호출량을 제한한다.

2

원리 네 가지

KV 캐시(계산→메모리) · PagedAttention(블록 할당) · continuous batching(빈자리 충전) · 양자화(정밀도→크기).

3

모델 선택

모델 지형과 MoE 구조, 규모별 실행 사양 기준.

4

앱 연결

base_url 교체만으로 7강 루프와 9강 앱이 로컬 모델 위에서 도는 구조를 확인했다.

5

오케스트레이터

LangGraph로 플래너·워커·종합을 구현하고, 리뷰 일괄 분석에 적용했다 (실측 검증).

정리
관리형 API의 제약(호출 한도·데이터 위치)은 자가 서빙으로 대체할 수 있는 선택 사항이다.
